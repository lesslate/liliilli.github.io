어플리케이션

기획서는 8월 23일까지 제출해야 하며, 내일 당장 핵심 인력들을 불러서 회의를 하고 싶지만 잘 안될 것 같다… 지금 닥친 문제가 한 두가지가 아니라서 좀 정리 좀 해보고 싶다.

기술력의 갭

대학교 동아리 특성 상 그냥 설렁설렁 하는 경우가 많다지만, 지금 일정이 빠듯한 관계로 일주일에 기능을 하나씩 구현할 수 있는 사람이 필요하다. 프로그래밍 인력 대충 6명 중에서, 안드로이드를 많이 다뤄봤고 기능 구현을 할 줄 안다는 사람을 스트릭하게 잡으면 한 두명 정도 될까 말까…

또한 Java 의 Exception 이라던가, Annotation 이라던가, 최소한 스레드가 뭔지는 알아야 안드로이드를 개발할 역량이 된다고 말할 수 있다고 개인적으로는 느끼는 바인데, OOP 가 뭔지도 모르는 사람들을 가지고 이끌려고 하면 대체 어쩌라는 건지 모르겠다. 내가 너무 이기적인가…

내가 후배한테 특강을 해서 잘 따라 온다면 후반부 투입이 될 수 있을지도 모르겠다. 그리고 미래의 인력으로 되기도 할 듯. 다른 사람들은 Google 이나 Duckduckgo 또는 StackOverFlow 에서 어떻게 검색해서 답안을 찾아내는 지만 알게 해줘도 목표는 대충 될 듯.

복잡성

Fragment 기반 화면

개발하고자 하는 프로젝트가 일반 Activity로 구현된 것이 아니라 Fragment 으로 대다수가 구현되어 있다. Fragment 는 Activity 에 비해서 좀 골치아픈 생명 주기를 가지고 있기 때문에 경우에 따라서 onPause() 등과 같은 콜백에서 잘 처리를 해줘야 버그가 일어나지 않는다던가 크래쉬가 일어나지 않게 된다.

onMeasure, onLayout, onDraw 가 사용될 가능성

매우 높을 것이라고 예상하고 있고, 1차 목표 선에서는 이 위의 무시무시한 콜백 메소드가 사용 될리는 없을 것 같지만, 2차 목표를 넘기기 시작하는 시점에서는 고려해야 할 리스트에 떡하니 있다.

라이브러리 사용법

나 혼자서 개발하는 프로젝트 같은 경우에는 나 혼자서만 사용법을 숙지해서 쓰면 된다지만, 다른 사람들에게 라이브러리 API 사용법을 숙지시키는 것도 문제이다. 그래서 잠시 생각해본 건데, 이렇게 쓰도록 하면 안될까?

  • 라이브러리 전반적인 이해는 내가 하고, 기본적인 사용법만 숙지시키도록 함.

그런데 사용될 라이브러리가 일반 대학교 학부생 프로젝트 선에서는 조금 많은 것이 흠이라고 해야할지, 일반 기업 프로젝트에서 쓰이는 라이브러리에 비해서는 1/3 수준인데.

리팩토링? 코드 스타일? 컨벤션?

리팩토링은 나중에 코드 유지 보수를 위해서도 필요하고, 다른 사람들이 코드를 잘 읽을 수 있도록 하는 관점에서도 중요한 코드 정리방법이기 때문에 어찌 굴러가던 간에 해야하는 것은 명백하다. 근데 이걸 나중에 한꺼번에 하기에는 너무 벅차기 때문에… 대충 이런 사이클로 리팩토링을 실시하면 좋지 않을까 싶다.

  • 기능 구현 -> 테스트 -> 부분 리팩토링 -> 테스트 -> 최종 업로드

코드 스타일은 전체적으로 통일하고 싶다.

  1. Tab은 Space 4개로 통일
  2. 브라켓은 K&R style을 고수할 것
