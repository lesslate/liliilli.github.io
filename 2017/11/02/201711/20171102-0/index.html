ctor 와 dtor 에는 virtual function 을 쓰지 말 것 (Effective C++ no.9)

Hexo
2017-11-02
C++, Effective_C++

요약

  • 생성자나 소멸자 안에서 virtual function 을 호출하지 말아야 한다. 가상 함수로서 virtual table 위에 적재가 된다고 하여도 지금 실행 중인 생성자나 소멸자에 해당되는 클래스의 파생 클래스 (Derived Class) 의 가상 함수가 호출되지 않는다.

Why?

다음 코드가 있다.

위 코드에서 b가 생성될 때, 상속에 의해 가장 기본 클래스인 Transaction 타입의 생성자가 호출된다. 그리고 파생 클래스로 밑으로 내려가면서 생성자가 호출되는 방식을 가진다.
하지만 기본 클래스에서 logTransaction()을 호출 할 때, 호출되는 함수는 BuyTransaction 의 함수가 아니라 Transaction 자체의 함수이다.

  • 기본 클래스의 생성자가 호출될 동안에는, 가상 함수는 절대로 파생 클래스 쪽으로 내려가지 않는다. 대신 생성자가 끝나고 파생 클래스로 내려가기 전까지는 객체 자신이 기본 클래스인 것 처럼 동작하게 된다.

만약 super의 생성자에서 derived의 함수로 내려갈 수 있다면, derived 클래스의 가상 함수는 해당 클래스의 멤버에 접근할 것이다. 그런데 아직 파생 클래스의 생성자에 의해 자신만의 변수가 초기화되지 않았으므로 Undefined Behavior 이 수행될 것이다.

그 외

  • RTTI (RunTime Type Information) 을 사용하는 모든 언어 요소를 생성자나 소멸자에서 사용할 때에도 위의 규칙을 따른다. dynamic_cast<>, typeid() 등..

  • 객체가 소멸될 때에는 역순으로 파생 클래스의 소멸자가 호출된 후에 기본 클래스의 소멸자가 호출되서 진행 될 때는 해당 객체의 런타입 타입 정보는 기본 클래스이다.

대처법

logTransaction() 함수를 Transaction 클래스의 비가상 멤버 함수로 바꾸고, 파생 클래스의 생성자들로부터 하여금 필요한 로그 정보를 Transaction 의 생성자로 넘겨서 호출하게 한다.

기본 클래스에서는 가상 함수를 따라 아직 초기화되지 않은 파생 클래스의 경계점을 넘어 접근할 수 없기 때문에, 역으로 필요한 초기화 정보를 파생 클래스에서 기본 클래스의 생성자로 올려보내서 해결할 수 있다.

Helper Method 에서 static 을 쓴 이유

만약 기본 클래스에 멤버 초기화 리스트가 많을 경우 혹은 BuyTransaction 객체의 미초기화된 데이터 멤버를 함부로 읽어 들이는 것을 방지하는데 도움이 되기 때문에 static을 넣어 설정할 수 있다.

  • 미초기화된 데이터 멤버는 정의되지 않은 상태에 있다.

위의 사실 때문에 기본 클래스에서 가상 함수를 호출 시, 파생 클래스로 내려갈 수 없게 된다.
